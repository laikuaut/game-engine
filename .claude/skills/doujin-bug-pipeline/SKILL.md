# doujin-bug-pipeline — バグ修正パイプラインスキル

報告されたバグを段階的に修正する。原因を特定・実証してから修正に入る。

## 事前チェック

1. `git status` で未コミット変更を確認し、ある場合はユーザーに警告
2. TodoWrite で8項目を登録

## パイプライン

### 1. バグ報告整理
ユーザーの報告から以下を整理する:
- **症状**: 期待動作 vs 実際の動作
- **再現手順**: どの操作で発生するか
- **影響範囲**: どの画面・機能に影響があるか
- **エラーログ**: コンソールエラーがあれば記録

不明な点があればユーザーに質問する。

### 2. 原因確認
関連コードを読み、データフローを追跡して原因の仮説を立てる:
- **何が**: どの変数・状態・処理が
- **なぜ**: どういう理由で
- **どこで**: どのファイルの何行目で

### 3. 原因実証
仮説が正しいことを実証する:
- ログ挿入、コード追跡、テスト再現のいずれかで確認
- 仮説が間違っていた場合は Step 2 に戻る（最大3回）
- 3回失敗したらユーザーに追加情報を求める
- **実証完了後、デバッグ用ログは必ず除去する**

### 4. `/doujin-design`
根本原因に対する修正方針を決定する。修正は最小限にとどめる。

### 5. `/doujin-implement`
設計に基づきバグを修正する。バグに無関係なリファクタリングは行わない。

### 6. `/doujin-desk-check`
修正をセルフレビュー。特に回帰リスクに注意する。

### 7. `/doujin-test-create`
**再現テスト必須** — 修正前なら失敗、修正後ならパスするテストを書く。

### 8. `/doujin-test-run`
全テストをパスするまで修正・リトライ（上限2回）。

### 9. `/doujin-doc-update`
通常不要。ドキュメント自体が間違っていた場合のみ修正。

## 完了報告

```
## バグ修正完了

### バグ報告
（元の症状説明）

### 根本原因
（原因の1行要約）

### 修正ファイル
- ファイル一覧

### テスト結果
- パス: X件 / 失敗: 0件
- 追加テスト: X件

### 回帰リスク
- （あれば）
```
